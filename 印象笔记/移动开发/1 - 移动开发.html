移动端的视口

viewport

直译过来叫做视口，这是一个虚拟的值，视口越来大，看到的内容就越多，注意这里与物理尺寸无关。

举一个例子：

通过设置 meta 标签的 content 属性，我们得到一个 viewport 为 500（左图）和 200（右图）的视口，图中方格的大小（px大小）一致。


可以看出左图视口大小为 500 的可以显示更多的内容。

所以这里得出一个结论：

相同物理尺寸下，viewport越大，看到的内容越多，内容越小。

在移动端下，如果不设置 viewport，默认大小为 980px，

设想一下，显示的内容会很多，但相应的内容也会变得非常小，这不是我们想看到的，所以移动端下都会设置 viewport 大小。

如果 viewport 设置太小，大屏手机内容会变太大，如果 viewport 设置太大，小屏手机的内容会变得太小看不清，

所以 viewport 的大小应该跟手机屏幕大小挂钩。

通常情况下，我们都会将 viewport 大小设置为 device-width，device-width 直译过来就是设备宽度，

设备宽度可以认为是手机开发商为我们手机量身定制的一个合理的 viewport，例如爱疯的375（移动端下，viewport 大小可以通过 screen.width 查看）。

这个 viewport 大小并非固定不变，我们可以在手机设置中修改手机显示大小改变它

比如我的手机一共有三个档位：


我们平常 css 所写的 px 是一个 viewport 单位

1px = 1 / viewport 宽

由于 viewport 可以改变，所以 px 是一个相对的值，又称为逻辑像素或是设备独立像素。

准确来说，px 是 viewport 像素（设备独立像素）。

dpr = 物理像素 / 设备独立像素

但有一个特殊的情况：

iphone6 plus的 viewport 宽为 414，通过 window.devicePixelRatio 可以得到： dpr 为3，

根据上面的公式可以得出：

物理像素=414 x 3=1242，

然而 iphone6 plus 的物理像素实际为 1080 x 1920，与计算出来的 1242 不一致，

关于这个问题，有一个解释是在逻辑像素和物理像素之间还有一层渲染像素，1242 就是渲染像素，在物理屏幕显示之前，需要经过一个叫做 downsampled 过程降低分辨率。

px 不是物理像素，不是逻辑像素，不是渲染像素，是与上述所有像素都无绝对逻辑关系的一个单位。

其实是浏览器内部对逻辑像素进行再处理的结果，简单来理解就是调整逻辑像素的缩放来达到适应设备的一个中间层。

layout viewport（布局视口）：

网页布局的基准窗口，相当于当前 PC 浏览器的窗口大小（不包括borders 、margins、滚动条），

如果把移动设备上浏览器的可视区域设为 viewport 的话，某些网站就会因为 viewport 太窄而显示错乱，

浏览器厂商为了让用户在小屏幕下网页也能够显示地很好，所以把布局视口宽度设置地很大，一般在 768px ~ 1024px 之间，大部分为 980 px。

在手机上，布局视口与移动端浏览器，屏幕宽度不再相关联，是完全独立的，这个浏览器厂商定的视口被称为布局视口（相当于 PC 专用视口）。

除了开发者设置缩放外，用户缩放不会影响布局视口


可以通过 document.documentElement.clientWidth/clientHeight 获取网页在标准模式下的布局视口大小。




布局视口，布局视口在之前有一个别名叫做初始包含块，而在比较早的文献中初始包含块也叫做画布。

理解画布可能比理解布局视口更简单。

如果你按比例绘图，很多时候就要参照你所用画布的大小，

比如设计师在 750px 画了一个 200px 的正方形，

如果你要在一张大小是 100cm 的纸上画，你可能就要这样计算正方形的宽度了 100cm * 200 / 750，

可以看到这个计算中是没有用到 dpr，你的笔触跨过多少个纸张分子，多少个原子根本就不影响我的绘图比例。

我们的画画的过程就相当于设置 css 的过程，css 的尺寸依赖的其实是布局视口的大小。


visual viewport（视觉视口）：

用户在移动设备屏幕上能看到的那部分区域。 

可以通过 window.innerWidth / innerHeight 来获取视觉视口大小。


理想视口(ideal viewport)：

理想视口是网站布局视口在移动端展示的理想大小。

使页面的宽度跟设备宽度一致（布局视口=理想视口 = 视觉视口），我们就能在移动屏幕上看到正常舒服的页面了。

当页面缩放比例为100%时，CSS像素 = 设备独立像素，理想视口 = 布局视口 = 视觉视口。

可以通过调用 screen.width / height 来获取理想视口大小。



移动端自适应

理解移动端布局主要是要理解布局视口的概念，还有浏览器是如何使布局视口匹配屏幕尺寸的，与 dpr 的关系不大。

自适应的目的：就是希望页面的设计与设计稿的设计比例一致，这个过程中针对不同的屏幕宽度元素的尺寸也会改变，但是一般不会有布局改变和元素的隐藏。

按照 css媒体查询 写的自适应严格来说不叫自适应，因为断点之间会造成比例误差,而让误差少一点就得多插值。很明显使用css媒体查询并不是做自适应的好方法。

基于大部分人还是很难理解 自适应 与 dpr 的关系，我特意再举一个通俗易懂的例子。

你想象你面前有一幅画，画里面画着一个女人和一个男人，他们相隔了一段距离。

然后你举起你的手机去拍摄这张画，为了让画的宽度刚好能显示在你的手机屏幕上，你开始移动你手机和画的距离，然后到合适了就按下拍摄按钮。

在这个过程中，画的宽度相当于布局视口，移动距离拍摄的过程相当于浏览器让布局视口匹配屏幕尺寸的过程，你的手机的拍摄像素相当于dpr。

你会发现就算你换不同像素的手机拍摄，画里面的男女的距离并不会走近或者拉远。

如果你和我说，我用高清屏拍摄发现男人和女人的距离在拉近，越高清距离拉得越近，那么你肯定撞鬼了。

真的，移动端尺寸自适应与dpr无关

淘宝方案

(function () {
  var dpr = window.devicePixelRatio;
  var meta = document.createElement('meta');
  var scale = 1 / dpr;
  meta.setAttribute('name', 'viewport');
  meta.setAttribute('content', 'width=device-width, user-scalable=no, initial-scale=' + scale +
  ', maximum-scale=' + scale + ', minimum-scale=' + scale);
  document.getElementsByTagName('head')[0].appendChild(meta);
  // 动态设置的缩放大小会影响布局视口的尺寸
  function resize() {
    var deviceWidth  = document.documentElement.clientWidth;
    document.documentElement.style.fontSize = (deviceWidth / 10) +'px';
  }
  resize();
  window.onresize = resize;
})()

width = device-width               先把布局视口放大 dpr 倍

initial-scale = scale = 1 / dpr   然后整体缩放相应倍数以适应设备尺寸


这是按照 640px 设计规范，设计图上标注 200px 元素大小，可以看出布局视口放大了 3倍，然后再整体缩放 1/3 到设备屏幕大小

200 / 640 = css像素大小 / 布局视口宽 = 351.563 / 1125 = 0.3125

由于这里是证明这个过程其实与 dpr 无关

我现在把 scale 的大小分别设置为 0.1

var scale = 0.1;

meta.setAttribute('name', 'viewport');

效果图


200 / 640 = 1171.88 /  3750 = 0.3125

我现在把 scale 的大小分别设置为 0.5

效果图

200 / 640 = 234.375 / 750 = 0.3125

结论：可以看到就算我设置 scale 不等于 1 / drp 的大小也不妨碍我按设计图的比例画出元素

1. scale 的值如果小于0.1布局视口也只能放大10倍，也就是布局视口最多放大10倍

2. 当 scale 的值大于1时布局视口并不会缩小，而且布局视口不再匹配设备宽度,这种情况实际不会出现。

看上面的截图我们看到字体大小发生了改变，在 scale 设置为 0.1 时基本就看不见了，

原因是一般我们的字体大小的设置不会使用 rem,而是使用 px 单位，这里的字体大小没有随布局视口的放大而增大，却会随页面的整体缩放而缩小

也有一些插件可以辅助把 px 转为 rem 的，但是方案是死的，人是活的，你只要把淘宝固有的十等份改一下就行了，比如设计稿是 640px 的，改一下

document.documentElement.style.fontSize = (deviceWidth / 6.4) +'px';

分了 6.4 等份

200px : 640px => xrem : 6.4rem

x = 2

流程： rem => 根元素字体大小 => 布局视口

其中一点就是这个方案可以很好地解决 1 px 边框的问题，对于高清屏来说设置 1 px 像素大小，其实横跨的是 dpr 个设备像素，这样看起来线条不够细，与设计稿就产生出入，

1px 问题

原因：设备像素比 dpr 与 1px 物理像素


公式：设备像素比(dpr) = 物理像素/设备独立像素（逻辑像素）。

如 iphone 6、7、8 的 dpr 为 2，那么一个设备独立像素便为 4 个物理像素，因此在 css 上设置的 1px 在其屏幕上占据的是 2个物理像素，

0.5px 对应的才是其所能展示的最小单位。这就是 1px 在 retina 屏上变粗的原因

viewport 的 initial-scale 具有缩放页面的效果。

对于 dpr=2 的屏幕，1px 压缩一半便可与 1px 的设备像素比匹配，这就可以通过将缩放比 initial-scale 设置为 0.5=1/2 而实现。

以此类推 dpr=3 的屏幕可以将 initial-scale 设置为 0.33=1/3 来实现。

解决方案

原理：利用 css 的 伪元素::after + transfrom 进行缩放

为什么用伪元素？ 因为伪元素::after或::before是独立于当前元素，可以单独对其缩放而不影响元素本身的缩放

注意：伪元素大多数浏览器默认单引号也可以使用，和伪类一样形式，而且单引号兼容性（ie）更好些

<div class="cell border-1px"> cell <div>

<style>
.cell {
    width: 100px;
    height: 100px;
}
<!--全部边框-->
.border-1px:after {
    content: '';
    position: absolute;
    box-sizing: border-box;
    top: 0;
    left: 0;
    width: 200%;
    height: 200%;
    border: 1px solid #000;
    border-radius: 4px;
    -webkit-transform: scale(0.5);
    transform: scale(0.5);
    -webkit-transform-origin: top left;
}

<!--单边框，以上边框为例-->
.border-1px-top:before {
    content: "";
    position: absolute;
    top: 0;
    left: 0;
    right: 0;
    border-top: 1px solid red;
    transform: scaleY(.5);
    transform-origin: left top;
}
</style>

移动开发中的像素

物理像素（设备像素）

像素是没有标准“大小”概念的单位（不像厘米、英寸、斤等单位），它不是一个确定的物理量，也不是一个具体的点或者小方块（尽管可以用点和小方块来呈现）。

关于像素有一个常见的错误理解：认为像素是一个宽高相等的小方块，并且的像素都是“那么大”，但是不知道这个宽高的具体数字。

我们描述的像素都是物理像素，即设备上真实的物理单元。

一个物理像素是显示器(手机屏幕)上最小的物理显示单元，在操作系统的调度下，每一个物理像素都有自己的颜色值和亮度值

下图是显示器屏幕的放大图，显示器屏幕实际上是由一个一个"点"组成的（每个"点"又包含3个单位，也称三元素组)，这些"点"就是物理像素。

反映的就是显示屏内部 led 灯的数量，可以简单理解，一组三色led代表一个物理像素，当然根据屏幕物理属性以及处理 led 的方法不一样。

强调这是物理的，因为这是一个纯硬件指标。比如我把屏幕锯了一半，物理像素就只有一半。

从屏幕在工厂生产出的那天起，它上面设备像素点就固定不变了，和屏幕尺寸大小有关，跟 Web 软件语言没一毛钱关系。


屏幕分辨率

任何设备的物理像素的数量都是固定的。比如 iPhone 5 的分辨率 640 x 1136 px，这表示手机分别在垂直和水平上所具有的像素点数。

当然手机屏幕分辨率高不代表屏幕就清晰，屏幕的清晰程度还与尺寸有关。

图片分辨率

比如一张图片的分辨率为 800 x 400。这表示图片分别在垂直和水平上所具有的像素点数为 800 和 400 。

同一尺寸的图片，分辨率越高，图片越清晰。





移动开发的分辨率：纵向像素点数 x 横向像素点数（图片也一样）

上图的标清手机屏幕分辨率：4 x 2 = 8

上图的高清手机屏幕分辨率：8 x 4 = 32

明显看出，高清手机屏幕的一个物理像素（设备像素）比标清手机屏幕的一个物理像素（设备像素）要小（相同面积下高清手机的物理像素更多）

所以，实际开发中不会使用物理像素（设备像素）



CSS 像素


CSS 像素：实际开发中写 CSS 时的像素（px），又叫逻辑像素或设备独立像素（density-independent pixel 即 dip / dp，意思是与设备无关的像素）

meta 里面设置 width=device-width，这个 device-width 就是设备独立像素。


在 chrome 里 看到的 375*667 等等都是设备独立像素，它们在数值上与 css 数值是相等的。实际 iPhone 的分辨率会比这高很多。

在写 CSS 时，我们用到最多的单位是 px，即 CSS 像素，当页面缩放比例为100%时，一个CSS像素等于一个 设备独立 像素


上图为CSS像素在标清手机屏幕和高清手机屏幕上的显示效果，CSS像素表示的大小是不变的，只是在高清手机屏幕上物理像素变多了

浏览器会自动将 CSS 像素推算成物理像素点数




设备像素比（dpr、 device pixel ratio）


当页面缩放比例为 100% 时，CSS 像素 = 物理像素，理想视口 = 视觉视口。

物理像素（设备像素、设计稿像素）和CSS像素（逻辑像素、设备独立像素）存在着一定的对应关系，又叫设备像素比，简称dpr，

dpr = 物理像素 / CSS像素

设计稿中，多少倍屏或者多少 x（1x，2x，3x）一般来讲就是指的 dpr 的数值。

#element { background-image: url('xxx.png'); }

@media only screen and (min-device-pixel-ratio: 2) {
    #element { background-image: url('xxx@2x.png'); }
}

@media only screen and (min-device-pixel-ratio: 3) {
    #element { background-image: url('xxx@3x.png'); }
}

设备像素比（dpr）：缩放比是1（100%）的情况下，单位尺寸内，在同一个方向（纵向或横向）的设备像素与 CSS 像素的比例

设备像素比（dpr）和当前的缩放状态有关。(通过控制整个值,可以控制页面的放大，快捷键Ctrl +的原理就是这个），

因为 CSS 像素是很容易被改变的，当用户对浏览器进行了放大，CSS像素会被放大，这时一个 CSS 像素会覆盖更多的物理像素，

页面的缩放系数 = CSS像素 / 物理像素 

设备物理像素(深蓝色背景)、CSS 像素(半透明背景)

当用户进行缩小操作时，一个物理像素覆盖了多个 CSS 像素：


当用户进行放大操作时，一个 CSS 像素覆盖了多个物理像素：


缩放是在放大或缩小 CSS 像素，不论我们进行缩小或放大操作，元素设置的 CSS 像素(如 width:300px )是始终不变的，

因为 CSS 像素被放大了，CSS 像素只是覆盖了更多的设备像素而已。

一个 CSS 像素对应多少个物理像素是根据当前的缩放比例（devicePixelRatio）来决定的。

拿到设计图如何布局？

设计师给的 750px 宽的设计稿是根据设备像素（device pixel，物理像素）为单位制作的设计稿。

前端工程师在编码 web 页面时所写的 CSS 像素 则需要根据 设备像素比 来进行换算。

首先，你要读懂设计师给你设计图的意图，一般国内的设计师，给出手机版的设计图，一般是750px，注意这里的px，并不是我们的 px（逻辑像素），其实是物理像素。

因为设计师是根据 iphone6 或者同等设备来进行设计的，那么要知道 iphone6 的逻辑像素其实是 375，dpr 是 2，那么拿到手的设计稿，转换为逻辑像素，就得除以 2，

我们叫这种设计图，叫两倍图。

同理，如果是375 + 375 + 375大小，那么我们就得除以3，叫三倍图。

如果设计师不用蓝湖等工具，给你的并不是 375 的倍数怎么办，我先说办法，原因你们自己琢磨，我不细致分析。

最简单的方法，设计师给你的图的物理宽度w，除以一个数x，如果得的出来的商在360 - 414之间，那么你换算的公式为【你在设计图测量出来的物理像素数除以 x 】，

那么 dpr 就是 x，也就是 x 倍图。

设备像素比（dpr）= 设备像素个数 / 理想视口CSS像素个数(device-width) ，则：

CSS 像素 = 设计稿像素（物理像素） / dpr（设备像素比）

所以我们开发过程中要以 CSS像素 * dpr 为准。

当 dpr=2 时，表示：1 个CSS像素 = 4 个物理像素，其实就是长和宽都增加了 2 倍。



只要设备像素比（dpr）> 1，就是高清手机屏幕，如果这样，css像素（设计稿上的像素） / 2 才可以获得与PC端相同的效果

因为 PC端 的设备像素比是 1

dpr 等于2的手机屏幕,设置 css 宽度为 1 px，其实覆盖的是 2 个物理像素

设备独立像素究竟是如何产生的：

智能手机发展非常之快，在几年之前，我们还用着分辨率非常低的手机，比如下面左侧的白色手机，它的分辨率是 320x480，我们可以在上面浏览正常的文字、图片等等。

但是，随着科技的发展，低分辨率的手机已经不能满足我们的需求了。很快，更高分辨率的屏幕诞生了，比如下面的黑色手机，它的分辨率是 640x940，正好是白色手机的两倍。

理论上来讲，在白色手机上相同大小的图片和文字，因为黑色手机的分辨率提高了一倍， 在黑色手机上会被缩放一倍。

这样，岂不是后面出现更高分辨率的手机，页面元素会变得越来越小吗？


然而事实并不是这样的，我们现在使用的智能手机，不管分辨率多高，他们所展示的界面比例都是基本类似的。

从 iPhone4 开始，乔布斯在 iPhone4 的发布会上首次提出了 Retina Display (视网膜屏幕)的概念，它正是解决了上面的问题，这也使它成为一款跨时代的手机。

之所以叫做视网膜屏幕，是因为屏幕的 PPI (屏幕像素密度)太高，人的视网膜无法分辨出屏幕上的像素点。

在视网膜屏幕（PPI特别高，一个屏幕上的物理像素点非常多）的设备上，如果一个CSS像素仍等于一个物理像素，那么网页上的各种元素就变得非常小，用户很难看清。

从苹果提出视网膜屏幕开始，才出现设备像素比这个概念，因为在这之前，移动设备都是直接使用物理像素来进行展示。

在 iPhone4 使用的视网膜屏幕会把 2x2 个物理像素当 1x1 个物理像素使用，这样让屏幕看起来更精致，但是元素的大小却不会改变。


如果黑色手机使用了视网膜屏幕的技术，那么显示结果应该是下面的情况

比如列表的宽度为 300 个 CSS 像素（设备独立像素），那么在一条水平线上，白色手机会用 300 个物理像素去渲染它，而黑色手机实际上会用 600 个物理像素去渲染它。


让2 个物理像素渲染 1 个设备独立像素只是 Retina 屏幕为了达到效果而使用的一种技术。

不是所有 DPR > 1 的屏幕就是 Retina 屏幕，比如：给你一块超大尺寸的屏幕，即使它的PPI很高，DPR也很高，在近距离你也能看清它的像素点，这就不算Retina屏幕。

移动端适配方案


流式布局当中的元素大小不会发生变化，不同设备的间距不一样

rem 布局是等比缩放，间距相同

当我们设置 viewport=device-width 的时候就已经可以让文字在移动端以合适的大小展示了，为什么还要做适配？

前面提到不同手机厂商对默认的 device-width 大小定义不一致，且用户可以修改 device-width 大小，而设计师不可能为每一个 viewport 都做一份设计稿，

通常情况下，设计稿的大小都是基于 iphone6：1334x750 ， dpr=2 产出。

为了让不同 device-width 的手机观看到的内容多少和设计稿一致，可以通过缩放的方法调和不同 viewport 造成的差异，这个就是 rem 适配方案。

而用 rem 替代 px 最根本的目的不是适配和缩放，是为了让 1 px是实现 1 物理像素。

根本的起源：

rem 替代 px 的原因是设计师要求 h5 跟 native 一样实现 border 的 1px，

作为设计师出生的前端开发，我是很能理解为什么有这个诉求的，

也是因为这个诉求完成以后，才让大家无法辨别页面是 h5 写的还是 native 写的；

简要的过程：

1px 的 border 本身还有其他替代方案，但要么存在兼容性问题（vw和0.5px），要么很难操作(background-image模拟)，

所以综合讨论下来觉得用 font-size + rem 的缩放方案替代原来的 px + view-port，让 px + view-port 的缩放单纯为物理像素服务，是最简单且兼容最全面的解法；

关键逻辑解析：

有两种模式“flex”和“vw”，都只是对font-size的计算规则不同，后者是真等比缩放，通常产品页面用flex，活动页面用全等比缩放的vw模式；

我们实际在 less 里面都不写 rem；而是通过打包工具把 px 换算成 rem。

换算逻辑你们如果仔细研究过的话会发现，我们特地把 1px 保留成 1px 输出，而不是 0.01rem 的方式输出；

所以，在 vw 和 0.5px 全兼容之前，rem 方案仍然可以用（但不推荐了）

https://www.zhihu.com/question/313971223/answer/1247564603

至于大屏看的更多而不是大屏看的更大的问题，

目前主流手机的 device-width 通常情况在 360～424 之间，横向有 60px 的内容差，纵向以 iphone6 和 iphoneX 比较，667 和 812 相差大概一个商品列表 item 的高度，

所以实际上大屏看的更多并没有多多少，忽略。

最后是适配方案，考虑兼容性的话用rem，不考虑可以用vw，其实目的都是为了做到等比缩放。

一套样式不可能适应各种大小的屏幕

针对不同的屏幕大小写样式

让我们的页面在不同大小的屏幕上都能正常显示

什么是栅格系统

栅格系统是响应式布局的实现方式


页面分为行和列，行和列的交汇处就是一个一个的格子，这些格子就是栅格，栅格与栅格之间是有空隙的，行与行也可以有空隙

栅格系统的列一般是固定的，只考虑列，不考虑行，有 12列、16列 和 24列的，上图一行分为12列

列为什么是固定的？因为行可以无限多，会有垂直方向的滚动条，列没有水平方向的滚动条，所以列会有固定的数目

每个格子都是结构，不是内容，响应式布局针对的就是这一个个的格子

一行不一定分为12列，一个格子可以占任意列，只要一行的格子占12列的宽度就可以（少的空出来，多的挤下去）


栅格系统的响应式布局原理：

改变栅格系统的格子所占列的宽度

一个图片从占据 1个格子 col-12 -> 2个格子 col-6 -> 3个格子 col-4 -> 4个格子 col-3 -> 6个格子 col-2

col-xl     超大屏

每行图片占据的格子宽度 = 12 / col-n = 12 /n

SassScript 支持 6 种主要的数据类型：

● 数字：1, 2, 13, 10px

$width: 10px;

● 字符串：有引号字符串与无引号字符串，"foo", 'bar', baz

$str: '3.jpg';

background-image: url('images/' + #{$str});

● 颜色：blue, #04a3f9, rgba(255,0,0,0.5)

$color: blue;

● 布尔型：true, false

● 空值：null

● 数组 (list)：用空格或逗号作分隔符，1.5em 1em 0 2em, Helvetica, Arial, sans-serif

$list: (100px, 200px, 3, '3.jpg');

div {
  width: nth($list, 1);
  height: nth($list, 2);
  zoom: index($list, 3);
}

● maps：相当于 JavaScript 的 object，(key1: value1, key2: value2)

$maps: (top: 1px, right: 2px, bottom: 3px, left: 4px);

div {
  top: map-get($map, top);
  left: map-get($map, left);
}

mixin在某些方面跟css类很像，最重要的区别是css类是在html文件中应用的，而mixin是在样式表中应用的。

判断一组属性是否应该组合成一个mixin，一条经验就是你能否为这个mixin想出一个好的名字。

如果你能找到一个很好的短名字来描述这些属性修饰的样式，比如rounded-cornersfancy-font或者no-bullets，那么往往能够构造一个合适的mixin。

如果你找不到，这时候构造一个mixin可能并不合适。
